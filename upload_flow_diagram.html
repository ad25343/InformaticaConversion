Upload Flow — Three Entry Paths into the Conversion Pipeline

Single mapping · Batch of mappings · Full project · All converge into the same 10-step processing pipeline

① Single Mapping Upload
One Informatica mapping at a time — interactive use
POST /convert · multipart/form-data
User uploads 1–3 files
Required: mapping XML
Required: etl_patterns YAML (config skeleton)
Optional: context document (PDF/DOCX — business rules)
↓
S0 Session Parse
route handler validates files
creates job record (status = PENDING)
dispatches single-mapping worker
↓
? ZIP or 3-file?
If user uploaded a .zip → zip_extractor unpacks
extracts XML + YAML + optional doc
If 3 raw files → used directly
↓
Single-Mapping Worker
Runs Steps 1–10 sequentially
One job_id tracks the entire run
Real-time status via GET /jobs/{id}
job = { job_id: uuid4(), status: "RUNNING", current_step: 1, state_json: gzip({}), complexity_tier: null → filled at S2 }
↓
GATE 1 Human reviews S2T + docs
Frontend shows parsed mappings, S2T table, docs
User approves, rejects, or amends inline
↓
GATE 2 Human reviews generated code
Shows converted files + security report
User can request changes → regen loop
↓
GATE 3 Final approval
Code review (Stage A/B/C) + perf report shown
User approves → export triggered
↓
✓ Output ZIP downloaded
Single mapping package:
main.py / model.sql s2t.yaml mapping_spec.md reviews.json
② Batch of Mappings Upload
Multiple mappings from one or more sources — parallel processing
POST /convert/batch · multipart/form-data
User uploads batch ZIP
ZIP contains N mapping XMLs
Each mapping has its own YAML config
(or a shared config is applied to all)
Optional: shared context doc
↓
S0 Batch Session Parse
zip_extractor unpacks archive
identifies N mapping XMLs
creates one batch_id + N job_ids
queues all jobs (status = PENDING)
↓
⚡ Parallel Orchestrator
Max 3 concurrent mapping workers
Semaphore prevents overload
Each worker is independent — one failure
does not block the others
async with semaphore: # max 3 await run_mapping(job_id) # Steps 1-10
↓
per mapping (in parallel)
S1–S5 Parse → Classify → S2T → Docs → Verify
Runs identically to single-mapping path
Results stored in each job's state_json
↓
GATE 1 (batch) Review all mappings' S2T
UI shows a mapping-by-mapping table
Reviewer can approve each individually
or approve all in one action
Failed mappings shown separately
↓
S7–S9 Stack Assign → Convert → Security → Code Review
Each mapping converted independently
Same parallel semaphore used
Stage A/B/C review per mapping
↓
GATE 2 + 3 (batch) Review code + final approval
Batch summary view: N approved, M failed
Reviewer can drill into individual mappings
Rejected mappings excluded from export
↓
Agg Batch Aggregator
Waits for all workers to finish or time-out
Collects completed vs failed counts
Merges approved outputs into one archive
{ "batch_id": "...", "total": N, "completed": K, "failed": N-K, "artifacts": ["job1.zip", "job2.zip", ...] }
↓
✓ Batch Output ZIP downloaded
One ZIP per approved mapping inside a
top-level batch archive:
mapping_1/ mapping_2/ … batch_summary.json
③ Full Project Upload
Entire Informatica PowerCenter project export — all mappings, workflows, params
POST /convert/batch · project ZIP OR manifest watcher
User uploads project ZIP (or drops in /watch dir)
Project export contains:
mappings/ (XML files) workflows/ parameter_files/ INDEX.txt manifest.json
Structure mirrors Informatica folder export
↓
S0 Project Session Parse
zip_extractor recursively unpacks
reads manifest.json → ordered mapping list
falls back to scanning all .xml files
assigns priority order (simple → medium → complex)
↓
? Manifest-driven or scan?
manifest.json found → use declared mapping list + metadata
No manifest → scan all XML files, auto-classify complexity
↓
Project Metadata Extraction
Reads global project config from manifest:
project_name, source_db, target_stack, folder_description, owner, etl_pattern overrides per mapping
Parameter files (dev/uat/prod) noted
Workflow dependencies noted (not converted)
↓
⚡ Parallel Orchestrator (project scope)
Same semaphore (max 3 concurrent)
Prioritises SIMPLE mappings first
(lower complexity → faster, unblocks reviewer sooner)
VERY_HIGH complexity flagged for human triage before conversion
↓
per mapping (in parallel, ordered by complexity)
S1–S5 Parse → Classify → S2T → Docs → Verify
Identical per-mapping pipeline
Project context doc injected as extra context
Cross-mapping field lineage noted (future)
↓
GATE 1 (project) Bulk S2T review — project dashboard
Project overview: complexity breakdown,
pattern distribution chart, per-mapping status
Reviewer can filter by pattern / tier
Approve tiers in bulk or individually
↓
S6–S9 Stack Assign → Convert → Security → Code Review
Stack determined per-mapping (can vary within project)
or set globally in manifest
All 3 review stages run per mapping
↓
GATE 2 + 3 (project) Code review + final sign-off
Project-level perf summary shown
(aggregate Stage C findings)
Reviewer signs off per-mapping or in bulk
↓
Exp Project Export Packager
Mirrors original project folder structure
Adds converted code alongside original XML
Generates project-level summary report
firstbank_converted/ mappings/ simple/ m_customer_load/ → main.py medium/ m_account_scd2/ → models/ complex/ m_gl_reconcile/ → ... project_summary.md batch_summary.json
↓
✓ Full project converted ZIP
Complete project with all approved mappings
N converted mappings project_summary.md batch_summary.json perf_report.json
Ready for deployment to target stack
